# 方案：订单页 D1 返工——严格按用户修正与老系统布局

> 日期：2026-07-05。状态：方案定稿，待 Codex 实施。
> 背景：D1(commit 5eaaea8) 用户完全不同意。用户逐条给出修正，核心是：
> D1 把三栏做出来了但①汇率放错位置②搜索内容退化③店铺筛选被全删④导出内容/顺序/物流位置不对。
> 本方案严格按用户 7 条修正 + 老系统 20251217 实际布局返工。以老系统 orderr/inc_list_default.php 为准。

## 用户修正逐条拆解（这是验收标准，不可偏离）

1. **实时汇率移出订单页** → 放到首页仪表盘（dashboard）。订单页中栏不再显示汇率。
2. **取消"信息"中栏** → 【搜索】占用原【信息】的位置（即搜索区扩大，不再是三栏挤搜索）。
3. **搜索内容按老系统各平台的搜索功能写全** → 不是 D1 那 10 个简短字段；
   引用"第一版"(commit 566905d)的丰富字段 + 老系统各平台字段（第一版已很接近，用户说第一版"直观好看"）。
4. **同步订单的"日数"移到租户【系统设置】** → 导出区的同步订单不再有天数下拉；
   天数改为在 settings 里配置，同步时读取该配置。
5. **导出区展示内容与顺序**（严格按用户给的，两两一组用 | 分隔）：
   【平台订单导入】【国际运单导入】｜【发货表导出】【平台订单表导出】｜【财务表导出】【客户资料导出】｜
   【发货通知表导出】【西阵发货表导出】｜【更新1688物流】【更新TB/PDD物流】｜【更新国际物流】
6. **店铺筛选两处被误删要恢复** → 用户要"像第一版那样展示"（第一版有店铺筛选）。
   ⚠️ 注意与上一轮"去掉筛选区店铺下拉"矛盾——以本次为准：**恢复店铺筛选，按第一版形态**。
7. **左侧只放"同步订单/导入导出"；物流更新三个按钮移到老系统的位置** →
   老系统里【更新1688物流】【更新TB/PDD物流】【更新国际物流】在**订单列表上方的批量操作工具栏**
   （orderr/inc_list_default.php:725-728、1040-1042 上下两处），不在筛选/导出区。

## 老系统实际布局（已核实，作为返工蓝本）

老系统 orderr 订单页从上到下：
- **筛选区**：左列搜索表单(约20项，见第一版字段) + 右列导入导出工具(常驻)。
- **列表批量工具栏**（列表正上方，勾选订单后操作）：全选/删除/设置订单状态/导出订单发货表(新)/
  推送飞兔/拉取飞兔/导出平台订单表/已发日本/下载订单图/**更新物流(1688)**/**更新物流(TB/PDD)**/
  **更新国际物流状态**/出荷处理表/财务核算表… ← 物流更新三按钮在这里。
- **订单列表** + 分页。

## 返工设计

### A. 汇率移到首页仪表盘

- 从 orders.php 中栏移除汇率显示；
- 在 dashboard 视图加实时汇率卡片（复用现有 `get_exchange_rate` / PriceCalculatorService::defaults
  的汇率取值），DashboardController 传汇率数据给首页视图。

### B. 订单页筛选区改为【左搜索(大) + 右导出工具】两栏（取消中栏）

- 布局从三栏改**两栏**：左栏搜索(占主要宽度)、右栏导出/物流工具（常驻大字，保留 D1 的常驻+大字优点）；
- 中栏原汇率/通告：汇率移首页(A)；通告(tenantNotices)移到搜索区上方或右栏顶部（保留可见，不丢）。

### C. 搜索内容恢复丰富字段（引用第一版 + 老系统）

- 恢复"第一版"(566905d) 的筛选字段集：基础区(订单号/1688单号/收件人/电话/邮箱/国内外单号/
  采购状态/签收地/每页) + 更多筛选折叠区(item_id/item_management_id/order_detail_id/lot_number(+empty)/
  kana片假名/product_name标题/pay_method/material/location/carrier/date范围/超时发货/已有运单等)；
- **店铺筛选恢复**（用户修正6）——按第一版形态放在搜索区（第一版有 store 相关，恢复它）；
- 平台差异仍留 D2 细化，但本次先把"字段丰富度"恢复到第一版水平（用户认可第一版直观）。
- OrderPageConfigRegistry 的 filterFieldsFor 扩充为第一版字段集（基础+高级两组）。

### D. 导出工具区按用户指定内容与顺序

exportToolsFor 输出改为用户给的清单，两两分组渲染：
```
组1: 平台订单导入 | 国际运单导入
组2: 发货表导出 | 平台订单表导出
组3: 财务表导出 | 客户资料导出
组4: 发货通知表导出 | 西阵发货表导出
组5: 更新1688物流 | 更新TB/PDD物流       ← 见下：物流更新归属
组6: 更新国际物流
```
⚠️ 但用户修正7又说物流更新三按钮要放到"列表批量工具栏"位置。两者需澄清——
**按老系统：物流更新在列表批量工具栏，导出区只放导入+各种导出。**
组5/组6的物流更新**移到列表批量工具栏**（见 E），导出区实际为组1-4：
```
平台订单导入 | 国际运单导入
发货表导出 | 平台订单表导出
财务表导出 | 客户资料导出
发货通知表导出 | 西阵发货表导出
```
同步订单单独一项（无天数下拉，天数读 settings）。
各导出指向现有控制器动作（发货表 /orders/export、平台订单表/西阵发货表/发货通知表/财务/客户资料
对应现有 ImportExportController 与 OrderExportController 动作），显示条件按现有权限。

### E. 列表批量工具栏加物流更新三按钮（老系统位置）

- 在订单列表正上方的批量操作工具栏（现有勾选批量区）加三个按钮：
  【更新1688物流】【更新TB/PDD物流】【更新国际物流】；
- 指向现有物流更新动作（/orders/logistics/update?type=1688 / express / jp，参照现有 updateLogistics
  的 type 参数）；权限沿用现有（1688物流/物流查看等）。

### F. 同步订单天数移到系统设置

- settings 加"平台订单同步默认天数"配置项（存 tenantSettings，如 profit 或新 section）；
- 导出区同步订单按钮不再有天数下拉，syncPlatformOrders 读该配置（无则默认7，保持兼容）；
- SettingsController 保存该项，OrderExportController::syncPlatformOrders 读取。

## 待用户确认的一处矛盾（实施前必须澄清）

用户修正5列了含物流更新的导出清单，修正7又说物流更新要放列表工具栏。
**本方案按修正7（老系统实况）处理：物流更新在列表批量工具栏，导出区不含物流更新。**
若用户其实想两处都有或另有想法，实施前调整。

## 不做 / 边界

- 平台差异化字段细节仍留 D2（本次先恢复到第一版丰富度）；
- 不改后端查询/权限；导出动作指向现有，不新建导出逻辑；
- 不动任务A悬停核价。
