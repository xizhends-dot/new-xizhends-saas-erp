# AGENTS.md — 西阵订单系统 · AI 编码代理执行准则

> 本文件对所有在本仓库工作的 AI 编码代理（Codex / Claude 等）生效。
> 项目技术约定见 [php-saas/CLAUDE.md](php-saas/CLAUDE.md)（命名空间/路由/视图转义/存储驱动等），
> 两者同时遵守；冲突时以任务提示词的明确指示为准。

## 编码四准则（源自 Karpathy LLM 编码守则，按本项目工作流适配）

### 1. 先想后写（Think Before Coding）

- 动手前把**假设显式写出来**；拿不准的直接停下来问，不要静默选一种理解做下去。
- 需求存在多种解读时，**列出解读让用户选**，禁止自行挑一种。
- 如果存在更简单的方案，必须说出来，哪怕与任务描述不同——允许有理有据地反对。
- 任务附带方案文档（php-saas/docs/plan-*.md）时，**先完整读方案再动手**，以方案为准。

### 2. 最简实现（Simplicity First）

- 只写解决当前问题的最少代码：不加没被要求的功能、不为单处使用建抽象、
  不加"以后可能用到"的配置项、不为不可能发生的场景写错误处理。
- 自问："资深工程师会不会说这段过度设计了？"是——就简化。
- 函数 <50 行、单文件 <800 行（全局规范）。

### 3. 外科手术式改动（Surgical Changes）

- **只动任务要求必须动的行**；每一行改动都应能追溯到任务要求。
- 不顺手"改进"相邻代码/注释/格式；不重构没坏的东西；风格跟随现有代码。
- 发现无关的疑似 bug / 死代码：**记录到小结，不要改**（会污染 diff，单独立项修）。
- 自己的改动产生的孤儿（不再使用的 import/变量/函数）要清理；既有的死代码不动。
- 结构性搬移任务（拆分/迁移）执行"纯搬家铁律"：方法体逐字移动，禁止任何顺手优化。

### 4. 目标驱动执行（Goal-Driven Execution）

- 动手前把任务转成**可验证的成功标准**（测试用例 / 冒烟检查点 / 验收清单），完成后逐条自验。
- 修 bug 先写能复现它的测试，再让测试通过；新功能先明确"怎么算做完"。
- 自检必做：php -l 改动文件 → 全量 `php php-saas/tests/*.php` → 登录态冒烟
  （**看渲染内容与数据落库，不要只看 HTTP 状态码**）→ 冒烟数据恢复。
- 任一自检失败：修到通过为止，不许带病提交。

## 本仓库红线（必守）

- **不要 git push**。是否 commit 按任务提示词的明确要求执行；未提及则不 commit。
- 视图所有输出走 `e()` 转义；所有 POST 表单带 `csrf_field()`。
- MySQL 是唯一生产驱动；JsonStore 只作演示/测试替身，但被 tests/*.php 使用的方法必须保持可用。
- `20251217/`、`old/`、`rust-backup/` 是只读参考资料，禁止修改。
- 冒烟用的临时数据（storage/data/app.json、测试订单/店铺）用完必须恢复/清理。
- 完成后输出结构化小结：改动文件、关键决策、自检结果、发现但未改的问题清单。
