# 西阵订单系统 — 重写设计方案

> 2026-05-14 讨论总结。基于现有 PHP 系统，用 Rust 重新开发为 SaaS 多租户订单管理系统。

## 一、技术栈

| 层 | 选型 | 理由 |
|------|------|------|
| 语言 | Rust | 性能、类型安全、内存安全 |
| Web 框架 | Axum | tokio 异步、类型安全路由、社区活跃 |
| 模板引擎 | Tera | 运行时渲染、改模板即刷新、类 Jinja2 语法 |
| 前端增强 | HTMX + Alpine.js（少量） | CDN 引入、无 npm 构建、声明式 AJAX |
| 数据库 | MySQL | 保持现有，只优化表结构 |
| ORM/查询 | SQLx | 异步、编译时 SQL 校验、灵活拼接动态查询 |
| 多租户 | 独立数据库 | 每公司一个库，隔离性最强 |

## 二、架构

### 2.1 三层架构

```
用户浏览器 ←→ Axum Web Server
               ├── 中间件：租户识别 → Session 认证 → 权限校验
               ├── 业务模块：订单/采购/物流/公司/员工/店铺/权限
               └── 数据层：主库（配置）+ N 个公司独立数据库
```

### 2.2 请求生命周期

```
请求 → 租户识别(域名/子域名) → Session 验证 → 权限检查(功能+店铺+操作级别) → Handler → Tera 渲染 → HTML
```

### 2.3 目录结构规划

```
src/
├── main.rs              # 启动入口
├── config.rs            # 配置
├── middleware/
│   ├── tenant.rs        # 租户识别
│   ├── session.rs       # Session
│   └── permission.rs    # 权限校验
├── models/              # 统一数据模型
│   ├── order.rs         # 订单（所有平台共用一套字段）
│   ├── store.rs         # 店铺
│   └── user.rs          # 用户/员工
├── handlers/
│   ├── auth.rs          # 登录
│   ├── dashboard.rs     # 仪表盘
│   ├── order_list.rs    # 订单列表
│   ├── order_detail.rs  # 订单详情
│   ├── order_save.rs    # 保存
│   ├── import.rs        # 导单
│   └── admin/           # 公司/员工/店铺管理
├── db/
│   ├── pool.rs          # 多租户连接池
│   └── queries.rs       # SQL
└── templates/           # Tera 模板 .html
    ├── layout.html      # 左右布局框架
    ├── dashboard.html   # 仪表盘
    ├── order_list.html  # 订单列表
    └── order_detail.html
```

## 三、多租户 SaaS 模型

### 3.1 角色层级

```
超管（你）
  ├── A公司 → 公司管理员 → 员工（客服/采购）
  ├── B公司 → 公司管理员 → 员工
  └── C公司 → ...
```

### 3.2 权限模型

三维组合控制：
- **功能模块**：导入、编辑、采购、导出、报表等分别授权
- **店铺范围**：控制每个员工能看到/操作哪些店铺
- **数据操作**：查看、编辑、删除

平台访问由超管在后台授权（tenant_platform 表）。

### 3.3 公告系统

- **系统公告**：主库存储，超管发布，全租户可见
- **公司公告**：租户库存储，公司管理员发布，内部可见

## 四、页面设计

### 4.1 整体布局

- 左侧深色侧边栏 220px（`#1F2937`）
- 右侧浅灰内容区（`#f9fafb`）
- 绿色强调色（`#07C160`）

### 4.2 菜单结构

```
首页仪表盘
平台订单
  ├── Yahoo 购物
  ├── 乐天 Rakuten
  ├── Wowma
  ├── Mercari
  ├── Qoo10（可锁定）
  └── 雅虎拍卖（可锁定）
采购
  └── 采购订单（跨平台汇总）
工具
  ├── 利润分析
  ├── 采购统计
  └── 全局搜索
管理
  ├── 公司设置
  ├── 员工管理
  └── 店铺管理
```

### 4.3 仪表盘

- 5 张统计卡片：待处理/今日采购/待发货/采购金额/物流异常
- 双栏公告：系统公告 + 公司公告
- 6 平台快捷入口
- 最近订单 + 操作动态

### 4.4 订单列表（四表结构）

每个订单 = 4 张表堆叠，共享 15 列统一网格：

| 区 | 表头色 | 内容 | 列数 | 平台订单 | 采购订单 |
|------|------|------|------|------|------|
| A区 | 浅紫 `#E0E7FF` | 客户信息 | ~13 | 可见 | 隐藏 |
| B1区 | 浅绿 `#D1FAE5` | 商品明细+采购状态 | ~14 | 可见 | 可见 |
| B2区 | 浅绿 `#D1FAE5` | 采购物流 | ~10 | 隐藏 | 可见 |
| C区 | 浅灰 `#F1F5F9` | 国际物流 | ~6 | 隐藏 | 隐藏 |

- A/C 区通过顶部「展开详情」按钮控制显示
- 操作日志独立按钮，以表格展示时间/操作人/操作类型/变更字段/变更详情/IP
- 采购状态放在订单时间右侧醒目位置

### 4.5 设计规范

- **全直角**：所有 `border-radius: 0`
- **统一边框**：`1px solid #9CA3AF`
- **表头对齐**：四表使用相同 `<colgroup>` 定义（c0~c14）
- **字体**：系统无衬线字体栈，订单ID 使用等宽字体
- **状态标签**：待处理(黄)/采购中(蓝)/运输中(绿)/已完成(灰)

## 五、关键设计要点

1. **统一字段**：6 个平台不再镜像目录，统一一套订单表结构和字段
2. **独立数据库**：每租户一个库，程序动态切换连接池
3. **平台订单 vs 采购订单**：两个视图共享同一数据结构，默认显示/隐藏不同区域
4. **操作日志**：记录每次字段变更（旧值→新值），追溯责任人
5. **大账号导单**：公司管理员统一导入，员工按店铺权限查看
